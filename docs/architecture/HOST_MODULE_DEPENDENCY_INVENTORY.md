# Host and Module Dependency Inventory

Date: 2026-07-17

Status: Phase 1D dependency baseline for
`docs/architecture/plans/HOST_AND_MODULE_BOUNDARY_IMPLEMENTATION_PLAN.md`.

## Purpose

This inventory records the current physical source-to-target graph and the hard dependency edges that
later host extraction and module splitting must remove or preserve deliberately. It describes the
repository as it exists before `GameHost`, `LayoutRealizer`, RuntimeUI decomposition, source moves, or
the final CMake target split.

The checked-in document explains the architectural consequences. The configured report generated by
CMake is the reproducible source of truth for exact translation-unit membership, direct link entries,
propagated link entries, include directories, and compile definitions in a particular platform
configuration.

The completed Phase 5A semantic owner map is recorded separately in
[`HOST_MODULE_FILE_CLASSIFICATION.md`](HOST_MODULE_FILE_CLASSIFICATION.md), with its exhaustive
machine-readable manifest in `cmake/NovelTeaModuleFileClassification.cmake`. This Phase 1D document
remains the pre-cutover physical dependency baseline rather than being rewritten as the desired graph.

## Reproducing the configured inventory

Configure any supported preset, then build the report-only target:

```sh
cmake --preset linux-debug
cmake --build --preset linux-debug --target module-dependency-inventory
```

The complete normalized report is written to:

```text
build/linux-debug/reports/module-dependency-inventory.txt
```

The report replaces the checkout and build roots with `<source>` and `<build>`, sorts targets,
translation units, and property values, and derives data from CMake target properties after all
repository subdirectories have been added. It includes targets with at least one first-party
translation unit; source-less interface targets and dependency-only targets remain visible through
the recorded link/include properties instead of being misclassified as NovelTea modules. It is
configuration-specific: Web, Android, devtools-off, and benchmark-enabled configurations should be
generated separately when those conditional edges are being changed.

The target is intentionally report-only in Phase 1. Phase 5 turns the desired graph into enforceable
module policy after source ownership already follows the intended direction.

## Known hard edges

### Flow and SessionState are inside `noveltea_core`

`noveltea_core` directly compiles:

- `engine/src/runtime/flow_executor.cpp` and the blockers/dialogue/interaction/Room/Scene execution files;
- `engine/src/core/session_state.cpp` and `engine/src/runtime/session_restore.cpp`;
- `engine/src/runtime/runtime_command_gateway.cpp`.

These are runtime implementation, not merely shared domain contracts. The target therefore cannot be
renamed wholesale to `noveltea_domain`. Phase 5 must separate stable values/contracts from execution,
session mutation, restore, and command-gateway implementation, with the latter moving to
`noveltea_runtime`.

### Runtime implementation and Lua are co-resident in `engine`

The `engine` target compiles `runtime/running_game.cpp`, all typed runtime executor files,
`runtime_session.cpp`, `runtime_checkpoint_service.cpp`, `script/lua/runtime_script_api.cpp`, the running-game
loader, and all sol2/Lua binding and `ScriptRuntime` files. It links Lua and sol2 privately, but the
runtime implementation has no physical target boundary preventing accidental Lua or host/backend
includes.

The split must establish `noveltea_runtime` before `noveltea_script_lua`: runtime owns abstract script
ports and execution; the Lua target implements those ports and depends inward on runtime/domain. The
running-game loader remains an approved content/host boundary rather than making JSON or asset loading
part of runtime execution.

### RuntimeUI crosses runtime, presentation, script, and backend seams

`engine/include/noveltea/ui_runtime.hpp` exposes or references `runtime::RunningGame`, typed runtime
input messages, `runtime::RuntimePublication`, Layout realization/policy values, SDL events, renderer
state, and optional `ScriptRuntime` initialization.

`engine/src/ui_runtime_rmlui.cpp` directly includes and uses:

- `ScriptRuntime` plus the private `script/lua/script_runtime_internal.hpp` state accessor;
- `RuntimeLayoutManager` and `RuntimePresentationBridge`;
- SDL, RmlUi Core/Lua, and sol2;
- RmlUi document, input, lifecycle, system, render, template, and custom-component internals;
- typed runtime input dispatch and runtime publication application.

This is the principal source-level hard edge behind the planned RuntimeUI facade decomposition.
Phase 2 removes runtime authority, Phase 3 moves Layout realization out, and Phase 4 removes preview
and playback ownership before target enforcement.

### Material and shader responsibilities are mixed into `noveltea_core`

`noveltea_core` compiles `render/material.cpp`, `material_codec.cpp`, `shader_compiler.cpp`, and
`shader_manifest.cpp` beside domain, runtime, presentation, and package code. Material value contracts
may remain in a backend-neutral domain surface where runtime consumers require them; material codecs,
shader manifests, shader compilation inputs, and package/export boundary behavior belong in
`noveltea_content` or build-host tooling. None of these files justify a dependency from domain or
runtime to bgfx.

### Editor protocol and JSON boundaries are physically broad

`core/editor_runtime_protocol.cpp` is compiled into `noveltea_core`. Public headers for compiled
package/project codecs, editor protocol, package export, rich-text codec, runtime settings codec,
save-state codec, material codec, and the running-game loader expose nlohmann JSON types. The target
links `nlohmann_json` privately, so direct boundary consumers such as tests, the editor tool, and the
benchmark must also declare JSON explicitly.

JSON implementation is additionally present in `Engine`, preview, app/resource-alias, running-game
loader, shader compiler, and test translation units. The final split must preserve the existing JSON
boundary policy: codecs and protocol adapters move to `noveltea_content`; runtime/domain/presentation
implementation must not gain JSON includes merely because they currently share a target.

### Android miniz and bimg share one canonical provider

`noveltea_content` owns ZIP package operations and privately links the canonical `miniz::miniz`
target. Desktop builds obtain that target from vcpkg. Android and Web builds, which use the NDK and
Emscripten toolchains rather than the vcpkg toolchain, provision the same pinned miniz release through
FetchContent.

Fetched bgfx.cmake builds are configured with that canonical target before bimg is added.
`cmake/patch-bgfx-miniz.cmake` removes TinyEXR's direct inclusion of vendored `miniz.c` and rewrites
TinyEXR's vendored include spelling to the canonical standalone header. This prevents both duplicate
`mz_*` providers and the previous header/implementation ABI mismatch. Final player/shared links must
contain exactly one standalone miniz provider. Android arm64-v8a and x86_64 and the Web link must be
revalidated after any change to this integration.

### `engine` publicly propagates concrete backends

The current `engine` interface publicly propagates:

- `noveltea_core`;
- SDL3;
- bgfx, bimg, bimg-decode, and bx;
- RmlUi Core;
- optional Dear ImGui;
- Android EGL/GLES, platform, logging, and OpenSLES libraries;
- backend/platform feature definitions and runtime asset-root definitions.

Lua, sol2, Twink, nlohmann JSON, miniaudio includes, text libraries, RmlUi Lua, and rmlui-bgfx are
currently private, although public Engine/RuntimeUI headers still expose enough concrete ownership to
force broad compilation coupling. The final `noveltea_engine` public contract should propagate only
usage requirements required to compile and link the backend-neutral facade.

### Tests compile against private engine source paths

The following targets receive `${CMAKE_SOURCE_DIR}/engine/src` as a private include directory:

- `noveltea_core_tests`;
- `noveltea_host_tests`;
- `noveltea_text_tests`;
- `noveltea_script_tests`;
- `noveltea_render_tests`;
- `noveltea_ui_tests`.

Current private-header consumers include host lifecycle/input/Layout/preview contracts, the Lua
runtime state accessor, RmlUi document/lifecycle/input/template/custom-component internals, and bgfx
material/shader/renderer internals. These tests are valuable characterization and must not be deleted,
but the blanket source-root grant prevents CMake from proving module encapsulation. During Phases 2-5
they must move with their owner, use target-scoped test-support headers, or test through an intentional
internal target rather than retaining repository-wide private visibility.

## Migration constraints derived from the inventory

| Constraint | First phase that may resolve it |
| --- | --- |
| Do not split targets while RuntimeUI still performs runtime dispatch or owns Layout realization. | Phases 2-3 |
| Do not make runtime depend on Lua/sol2; introduce the runtime target before the Lua adapter target. | Phase 5A/5B |
| Keep JSON and editor protocol implementation in approved content/boundary owners. | Phase 5A/5C |
| Preserve exactly one miniz implementation provider in Android final links. | Phase 5C and platform validation |
| Remove public backend propagation only after Engine uses a private implementation boundary. | Phase 2D, finalized in Phase 5E/6A |
| Retarget private-source tests only after the internal owners and test seams exist. | Phases 2-5 |

## Phase 1D conclusion

The current graph's difficult edges are physical rather than unresolved semantic contracts. The
target owners, extraction order, JSON boundary, runtime-to-Lua direction, Android miniz constraint,
and private-test migration obligations are now explicit and reproducible. No production source move,
target split, or backend-link change is part of this phase.
